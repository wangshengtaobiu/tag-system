# Ontology Factory 操作手册

> 本工程用于将原始标签目录梳理为标准化的冻结本体。
> 架构和设计说明见 `ontology_factory_design.md`，框架说明见 `ontology_factory_framework.md`。

---

## 一、环境搭建

| 要求 | 说明 |
|------|------|
| Python | **3.10+** |
| 操作系统 | Windows / Linux 均可 |
| 网络 | 能访问 `api.deepseek.com` |

```bash
# 进入工程目录
cd ontology_factory

# （推荐）创建虚拟环境
python -m venv venv
venv\Scripts\activate      # Windows
source venv/bin/activate   # Linux
```

---

## 二、依赖安装

整个流水线只需要两个第三方包：

```bash
pip install requests pyyaml
```

| 包名 | 用途 | 不装后果 |
|------|------|----------|
| `requests` | 调用 DeepSeek API | S2/S3/S4/S5/S8 等所有 LLM 阶段无法运行 |
| `pyyaml` | 读取 `factory_config.yaml` | 配置加载回退 JSON 模式，可忽略 |

**不需要任何本地模型、GPU、PyTorch、transformers 等**。流水线全走 DeepSeek 云端 API。

---

## 三、依赖的 AI（全部云端）

| 角色 | 模型 | 用途 |
|------|------|------|
| **Flash (Worker)** | `deepseek-v4-flash` | 批量标准化（S2）、ID 生成辅助（S4）、别名检测（S5） |
| **Pro (Architect)** | `deepseek-v4` | 命名空间设计审查（S3）、ID 冲突仲裁（S4）、生产冻结（S8） |

两者都通过 DeepSeek API 调用，不需要本地部署。

## 四、人工配置

### 4.1 API Key

```bash
# 设环境变量（推荐）
set DEEPSEEK_API_KEY=sk-xxxxxxxxxxxxxxxx        # Windows
export DEEPSEEK_API_KEY=sk-xxxxxxxxxxxxxxxx     # Linux

# 或者直接写在 config/factory_config.yaml 里：
#   models:
#     pro:
#       api_key: "sk-xxxxxxxxxxxxxxxx"
#     flash:
#       api_key: "sk-xxxxxxxxxxxxxxxx"
```

API Key 申请：https://platform.deepseek.com

### 4.2 领域配置 `profiles/adult_profile.json`

这是唯一的领域相关文件，定义：
- **命名空间**（25 个，如 `sexual_behavior`、`fetish_preference` 等）及其映射关系
- **语义类型**（标签的分类粒度，如 `body_anatomy`、`sexual_act`）
- **本体类型规则**（flat_behavior / specialized_behavior / graph_native / meta_style 的判定阈值）
- **别名策略**（合并置信度阈值、审查阈值）

**换领域时需重写此文件。** 编写指南见 `ontology_factory_design.md` 第三部分。

### 4.3 流水线配置 `config/factory_config.yaml`

需要人工确认的关键项：

```yaml
# API 地址（默认不用改）
models:
  pro:
    api_base: "https://api.deepseek.com/v1"
    model: "deepseek-v4"
  flash:
    api_base: "https://api.deepseek.com/v1"
    model: "deepseek-v4-flash"

# 置信度阈值（已冻结，勿改）
confidence:
  auto_accept: 0.85       # ≥ 此值直接通过，无需人工审核
  needs_review: 0.70      # 此值到 0.85 之间标记为需审查
  reject: 0.60            # < 此值拒绝

# 批次大小
pipeline:
  stages:
    s2_normalize:
      batch_size: 50      # 每批送 LLM 的标签数，可按需调整
```

### 4.4 提示词模板 `prompts/`

| 文件 | 对应阶段 | 用途 |
|------|----------|------|
| `s2_normalize_prompt.md` | S2 | 标准化提示词 |
| `s4_freeze_prompt.md` | S4 | ID 冻结提示词 |
| `s5_alias_prompt.md` | S5 | 别名合并提示词 |

提示词由架构师维护，运行前无需修改。

## 五、原始数据输入格式

数据入口在 S1 triage（`s1_triage.py`），它是唯一读取原始输入的地方。S1 将多种 key 变体统一为内部标准字段名，后续阶段（S2~S8）只读标准化后的字段。

### 5.1 可接受的字段名

S1 triage 接受中英文 key 混用，每个字段支持以下变体：

| 标准化字段 | 可接受的原始 key | 必填 | 说明 |
|-----------|-----------------|------|------|
| `name` | `name` / `标签名` / `original_name` | **是** | 标签名 |
| `category` | `category` / `分类建议` / `分类` | 否 | 原始所属分类（2~4 字主类名） |
| `definition` | `definition` / `定义说明` / `描述` | 推荐 | 标签语义定义，60~200 字 |
| `distinction` | `distinction` / `区别` | 否 | 与相似标签的核心差异 |
| `parent_name` | `parent_name` / `上位tag` | 否 | 上位（父）标签名，无则留空 |
| `examples` | `examples` / `示例词` / `文学示例词` / `示例` | 否 | 示例词数组或逗号分隔字符串 |
| `heat` | `heat` / `热度` | 否 | 热度数值（仅 S1 统计用，后续阶段不消费） |

### 5.2 JSON 格式（推荐）

```json
[
  {
    "name": "口交",
    "category": "核心性行为",
    "definition": "以口唇、舌部刺激对方生殖器的性行为方式",
    "distinction": "与舔棒的区别：口交侧重含入包裹，舔棒仅表舌面接触",
    "parent_name": null,
    "examples": ["含入", "舐弄", "口内射精"],
    "heat": 5000
  },
  {
    "标签名": "肛交",
    "分类建议": "核心性行为",
    "定义说明": "以肛门为插入部位的性行为方式",
    "热度": 3500
  }
]
```

> JSON 支持中英文 key 混用，S1 triage 自动识别。数组内每条是一个标签对象。

### 5.3 命名与路径

输入文件名任意，通过 `--input` 参数指定路径。

## 六、运行

### 6.1 基础命令

```bash
cd ontology_factory

# 完整流水线
python run_factory.py --profile profiles/adult_profile.json --input raw_tags.json

# 从指定阶段开始
python run_factory.py --profile profiles/adult_profile.json --input raw_tags.json --stage s4

# 到指定阶段停止
python run_factory.py --profile profiles/adult_profile.json --input raw_tags.json --end-stage s6

# 干跑（仅校验输入不执行）
python run_factory.py --profile profiles/adult_profile.json --input raw_tags.json --dry-run
```

### 6.2 全部参数

| 参数 | 必填 | 默认值 | 说明 |
|------|------|--------|------|
| `--profile` / `-p` | **是** | — | 领域配置 JSON |
| `--input` / `-i` | **是** | — | 原始标签文件（JSON） |
| `--config` / `-c` | 否 | `config/factory_config.yaml` | 工厂配置 |
| `--stage` / `-s` | 否 | `s1` | 起始阶段 (s1~s8) |
| `--end-stage` / `-e` | 否 | `s8` | 结束阶段 (s1~s8) |
| `--work-dir` / `-w` | 否 | `./work` | 中间产物目录 |
| `--exports-dir` | 否 | `./exports` | 最终输出目录 |
| `--dry-run` | 否 | — | 仅校验，不执行 |

### 6.3 8 阶段流水线

```
S1 目录分拣(脚本) → S2 语义标准化(Flash) → S3 命名空间(Pro) → S4 ID冻结(Flash+Pro)
→ S5 别名合并(Flash+Pro) → S6 验证审计(脚本) → S7 检索导出(脚本) → S8 生产冻结(Pro)
```

| 阶段 | 执行者 | 输入 | 输出 | 是否需要 API |
|------|--------|------|------|-------------|
| S1 目录分拣 | 脚本(自动) | raw_tags.json | work/inventory_clean.json | 否 |
| S2 语义标准化 | Flash API | S1 输出 | work/stage2_normalized.json | **是** |
| S3 命名空间审查 | Pro API | S2 输出 + profile | work/namespace_freeze.json | **是** |
| S4 ID 冻结 | Flash+Pro API | S2+S3 输出 | work/stage4_resolved.json | **是** |
| S5 别名合并 | Flash+Pro API | S4 输出 | work/stage5_aliases.json | **是** |
| S6 验证审计 | 脚本(自动) | S5 输出 | work/s6_audit_report.json | 否 |
| S7 检索导出 | 脚本(自动) | S6 输出 | work/retrieval_index_pre.json | 否 |
| S8 生产冻结 | Pro API | 全部中间产物 | exports/* | **是** |

### 6.4 断点续传

流水线支持从任意阶段开始。例如上次 S4 出错后可：

```bash
python run_factory.py --profile profiles/adult_profile.json --input raw_tags.json --stage s4
```

S1~S3 的中间产物需保留在 `work/` 目录下。

## 七、输出位置

### 7.1 最终产物 `exports/`

```
exports/
├── ontology_export_v1.json    # 冻结的规范本体（主产物）
├── retrieval_index.json       # 检索索引
├── alias_graph.json           # 别名 → 主条目映射
└── freeze_manifest.json       # 版本记录
```

| 文件 | 说明 |
|------|------|
| `ontology_export_v1.json` | 每条标签的最终版本：canonical_id、namespace、semantic_type、definition、aliases、confidence 等全部字段 |
| `retrieval_index.json` | 供下游打标系统使用的检索结构 |
| `alias_graph.json` | 别名到主条目的解析映射 |
| `freeze_manifest.json` | 冻结时间戳、数据指纹，用于版本审计 |

### 7.2 中间产物 `work/`

```
work/
├── inventory_clean.json        # S1: 去重后的干净清单
├── s1_stats.json               # S1: 统计报告
├── stage2_normalized.json      # S2: 标准化结果 + 置信度
├── stage2_review_queue.json    # S2: 需人工审查的低置信度条目
├── namespace_freeze.json       # S3: 冻结的命名空间
├── stage4_resolved.json        # S4: ID 去重后的本体
├── stage5_aliases.json         # S5: 别名提案
├── s6_audit_report.json        # S6: 9 项检查结果
└── review_queue.json           # 汇总审查队列
```

### 7.3 快速验证

```bash
# 查看本体条目数
python -c "import json; d=json.load(open('exports/ontology_export_v1.json')); print(len(d))"

# 检查是否有重复 canonical ID（必须为 0）
python -c "
import json
from collections import Counter
d = json.load(open('exports/ontology_export_v1.json'))
cids = Counter(e['canonical_id'] for e in d if not e.get('is_alias_of'))
dupes = {k:v for k,v in cids.items() if v>1}
print('重复ID:', len(dupes), list(dupes.keys()) if dupes else '✓ 无')
"
```

## 八、常见问题

| 问题 | 解决 |
|------|------|
| `ConnectionError` / `401` | 检查 API Key 是否正确、DeepSeek 服务是否正常 |
| JSON 解析失败 | LLM 返回格式异常，会自动重试 3 次并降级 batch_size |
| 某批失败后流水线中断 | 减小 `factory_config.yaml` 中的 `batch_size` 后重跑 |
| 编码乱码 | JSON 文件确保 UTF-8 编码，保存时使用 `ensure_ascii=False` |
| PyYAML 未安装警告 | `pip install pyyaml`，或忽略（自动回退 JSON 解析） |
| 重复 canonical ID 不为 0 | 重新跑 S4，确认 Pro 仲裁已覆盖全部冲突 |
